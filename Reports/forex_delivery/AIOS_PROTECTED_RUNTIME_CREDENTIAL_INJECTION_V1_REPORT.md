## Runtime Injection Behavior

The new module provides:

- `build_runtime_injection_contract`
- `validate_runtime_injection_request`
- `build_runtime_auth_provider`
- `build_sanitized_runtime_injection_summary`
- `classify_runtime_injection_blockers`

The contract requires authenticated operator state, protected action authorization, live exception intent, live risk acknowledgement, live runtime approval, runtime-only credentials, no credential or account persistence, one network allowance, one-trade-only, micro-size-only, no retry behavior, no repeated behavior, max order count of one, injected auth provider, injected HTTP client, final bridge readiness, OANDA transport readiness, and OANDA connector readiness.

`build_runtime_auth_provider` accepts injected access token and account id values, rejects blanks, and returns a callable provider. The sanitized contract and summary report only whether a provider exists; they do not return credential or account identifier values.

## Protected Local Harness Behavior

`build_protected_local_execution_harness` builds local readiness evidence from the injected request, HTTP client, runtime auth provider, and order intent.

Dry-run mode is the default and does not call `http_client.post`. Non-dry-run mode still blocks unless `protected_live_execution_command` is explicitly true. The harness returns sanitized summary, blockers, safety flags, and readiness status only. It does not place an order.

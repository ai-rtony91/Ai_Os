## Core Findings

1. The canonical hierarchy is compact and already sufficient.
   PRG-FOREX-001 defines exactly one program, four epics, eight buckets, and ten packet anchors. No new hierarchy doctrine is needed.

2. The report layer is the main source of complexity.
   Reports/forex_delivery had 508 files before this report, while the canonical hierarchy has only 13 files. Most complexity is not in governance law; it is in generated evidence, repeated runbooks, packet drafts, and status reports.

3. Most active Forex reports are not linked to PRG/EPC/BKT/PKT identity.
   Only two files in the active Forex delivery/orchestration/trading docs searched referenced PRG-FOREX-001, EPC-FOREX, BKT-FOREX, or PKT-FOREX: AIOS_FOREX_EPIC_BUCKET_PACKET_CONSOLIDATION_V1_REPORT.md and AIOS_FOREX_SUPERVISED_DEMO_OPERATIONAL_VALIDATION_RUNNER_V1.md.

4. Reports are being named like packets but are not governed packets.
   Many report filenames contain packet-like language such as Packet K, Packet L, Packet M, Packet O, Packet P, Packet Q, Packet R, Packet S, Packet T, Packet U, exception packet, execution packet, and approval packet. The canonical program declares only PKT-FOREX-001 through PKT-FOREX-010 as anchors. These report names need classification as evidence, draft, archive, or future-packet proposal.

5. OANDA/demo/live/profit evidence chains overlap heavily.
   OANDA, broker demo, live micro-trade, profit proof, P/L, read-only, and manual-finalization reports describe repeated stages of the same proof chain. This should be consolidated into indexes and canonical result summaries, not new doctrine.

6. docs/forex_delivery, docs/trading_lab, and docs/orchestration each contain partial Forex operating maps.
   The three active doc roots overlap on live readiness, OANDA demo, paper execution, risk, dashboard truth, evidence, and strategy validation. Each root should have one role: delivery gate docs, builder docs, or orchestration docs, with cross-links to the PRG/EPC/BKT hierarchy.

7. Existing source-of-truth policy already classifies many legacy areas.
   docs/governance/source-of-truth-map.md says docs/AI_OS is CLEAN-era source material pending merge/archive classification and archive/** is historical/reference-only. Forex consolidation should not revive archive files as live authority.

8. Version chains create stale-reader risk.
   Examples include V10/V11 capital-flow files, broker demo V2-V6/V7-V9 files, repeated first-demo-connection-proof files, and paired base/manual-finalization files. These should be indexed with one current result per chain.

9. Live trading remains blocked by root policy.
   RISK_POLICY.md makes live trading, broker execution, OANDA/live order execution, and credential handling blocked by default unless a specific Single Live Micro-Trade Exception is active and fully scoped. The Forex consolidation should preserve this root boundary.

10. The highest-value consolidation is operator-facing, not raw evidence deletion.
    Evidence files may need to remain for audit history. The simplification should reduce what an operator or worker reads as current guidance by creating/using indexes, archive labels, and canonical closeout pointers.

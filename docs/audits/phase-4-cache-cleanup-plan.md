# Phase 4 Cache Cleanup Plan

## Purpose

Phase 4 is a cache-only cleanup plan based on Phase 3 dependency validation.

This plan does not approve deletion by itself. It records the exact cache cleanup commands to run only after explicit user approval.

## Cleanup scope

Scope is limited to generated cache/vendor artifacts only:

- pytest cache
- Python `__pycache__` folders
- dashboard installed vendor dependencies

No source, runtime, launcher, orchestration, Trading Lab, telemetry, authority, archive, packet, worker, approval, or control-script paths are in scope.

## Eligible cleanup targets

| Target | Classification | Reason | Risk | Approval required |
|---|---|---|---|---|
| `.pytest_cache/` | REMOVE/RELOCATE CANDIDATE | Generated pytest cache; no active dependency required | low | YES |
| `**/__pycache__/` | REMOVE/RELOCATE CANDIDATE | Generated Python bytecode/cache folders | low | YES |
| `apps/dashboard/node_modules/` | REMOVE/RELOCATE CANDIDATE | Installed vendor dependencies; reinstallable if package files remain | medium | YES |

## Approved candidates

These are approved as candidates for a future cleanup command only, not approved for deletion in this document:

## Pre-cleanup validation

Run these before any approved cleanup:

```powershell
git status --short
Test-Path .pytest_cache
Get-ChildItem -Path . -Directory -Recurse -Force -Filter __pycache__ | Select-Object FullName
Test-Path apps\dashboard\node_modules
Test-Path apps\dashboard\package.json
Test-Path apps\dashboard\package-lock.json
```

Stop if:

- `apps/dashboard/package.json` is missing.
- `apps/dashboard/package-lock.json` is missing and the user does not approve reinstall risk.
- `git status --short` shows unexpected changes outside planning docs.
- Any command points outside `C:\Users\mylab\OneDrive\GitHub\AI_OS_V2`.

## Exact proposed cleanup commands

Run only after explicit user approval.

Cache-only cleanup:

```powershell
Remove-Item -LiteralPath .pytest_cache -Recurse -Force -ErrorAction SilentlyContinue
Get-ChildItem -Path . -Directory -Recurse -Force -Filter __pycache__ |
  Where-Object { $_.FullName -notmatch '\\.git|\\.codex_worktrees|archive|node_modules' } |
  Remove-Item -Recurse -Force
```

Optional dashboard vendor cleanup:

```powershell
Remove-Item -LiteralPath apps\dashboard\node_modules -Recurse -Force
```

Do not run the optional dashboard vendor cleanup unless the user confirms reinstall is acceptable.

## Post-cleanup validation

Run after approved cleanup:

```powershell
git status --short
git diff --check
Test-Path .pytest_cache
Get-ChildItem -Path . -Directory -Recurse -Force -Filter __pycache__ |
  Where-Object { $_.FullName -notmatch '\\.git|\\.codex_worktrees|archive|node_modules' } |
  Select-Object FullName
Test-Path apps\dashboard\node_modules
```

Expected result:

- `.pytest_cache/` absent if approved.
- `__pycache__/` folders absent outside excluded areas if approved.
- `apps/dashboard/node_modules/` absent only if optional vendor cleanup was approved.
- No source, runtime, orchestration, dashboard source, Trading Lab, telemetry, packet, worker, approval, launcher, service, or control-script files changed.

## Rollback notes

- `.pytest_cache/` can be regenerated by running pytest.
- `__pycache__/` folders can be regenerated by Python execution.
- `apps/dashboard/node_modules/` can be restored with dependency install from `apps/dashboard/package.json` and lockfile.
- If package restore is needed, request approval before running network/dependency install commands.

## Rollback/reinstall notes

Cache cleanup is low risk because caches regenerate. Dashboard vendor cleanup has higher operational risk because it may require reinstalling dependencies.

If `apps/dashboard/node_modules/` is removed and the dashboard needs to run later, reinstall from inside `apps/dashboard/` using the project package files after explicit user approval for dependency installation.

## User approval required

Required before any cleanup command:

- approval to delete `.pytest_cache/`
- approval to delete `__pycache__/` folders
- separate approval to delete `apps/dashboard/node_modules/`
- confirmation that dashboard dependency reinstall is acceptable before vendor cleanup

No commit or push is included in Phase 4.

## Risks

- Removing `apps/dashboard/node_modules/` may temporarily prevent local dashboard dev/build commands until dependencies are reinstalled.
- Removing caches may change local timestamps and regenerate files during later Python/test runs.
- A broad recursive delete command is risky if filters are wrong; commands must preserve `.git/`, `.codex_worktrees/`, `archive/`, and `node_modules/` except the separately approved dashboard vendor folder.
- No runtime, orchestration, telemetry, Trading Lab, dashboard source, or authority files should appear in `git status` as modified after cache cleanup.

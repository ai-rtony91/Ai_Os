# AIOS Closed Autonomy Loop

## Purpose

The Closed Autonomy Loop v1 is a one-cycle control-plane report builder for
AI_OS. It composes local evidence, the Autonomy Decision Governor, proposed
packet shape, validation gate expectations, dispatch recommendation, and stop
state into one deterministic report.

The cycle is:

```text
Observe -> Decide -> Plan -> Validate Gate -> Recommend Dispatch -> Report -> Stop
```

## Why It Exists

The Autonomy Decision Governor ranks the next safest highest-value task. The
closed loop uses that decision as an input and answers the next operational
question: what would one governed cycle recommend now, with evidence and safety
boundaries attached?

This moves AI_OS from ranking work toward completing a single governed autonomy
cycle. The output is still evidence and recommendation, not execution.

## Relationship To The Governor

The governor remains the owner of candidate ranking and next-task selection.
The closed loop does not duplicate ranking logic. It either consumes the latest
governor JSON or calls the governor functions in memory, then normalizes the
selected candidate into a proposed cycle action.

The closed loop owns:

- loop phase status
- proposed cycle action shape
- conservative gate result
- dispatch recommendation with dispatch disabled
- one-cycle stop evidence

## Why v1 Does Not Dispatch Workers

Worker dispatch is a protected autonomy boundary. Dispatch requires approval,
lock, queue, worker-state, validator, and stop-point integration that is broader
than this v1 packet.

Closed Autonomy Loop v1 emits:

- `dispatch_authorized: false`
- recommended next packet type
- recommended next packet ID
- recommended worker lane
- whether human action is required

It does not launch workers, mutate queues, write inbox state, call providers,
start schedulers, or run background loops.

## Latest Report Materializer

The closed loop can materialize its one-cycle report to:

```text
Reports/sandbox/closed_autonomy_loop/AIOS_CLOSED_AUTONOMY_LOOP_LATEST.json
```

This sandbox latest report is generated by the closed-loop owner itself. It is
evidence for downstream packet drafting and gate previews, not queue authority.
The materialized report preserves:

- one-cycle only mode
- `dispatch_authorized: false`
- `queue_mutation_authorized: false`
- `continuous_loop: blocked`
- conservative missing-evidence handling

No report materialization step dispatches workers, mutates queues, starts a
scheduler, touches credentials, calls brokers, or executes webhooks.

## Why v1 Is One-Cycle Only

Continuous operation changes the risk profile. A repeated loop needs durable
Start, Pause, Resume, Stop, watchdog, telemetry, idempotency, retry, lock, and
operator override controls.

This v1 always stops after one evaluation. It can be called repeatedly by a
future approved controller, but it does not become that controller.

## Start, Pause, Resume, Stop Controls

Start, Pause, Resume, and Stop controls become appropriate only after:

- one-cycle reports are stable
- dispatch remains gated and observable
- queue and worker-state locks are integrated
- validator repair paths are deterministic
- approval evidence is explicit and scoped
- scheduler and background loop behavior are separately approved

Those controls belong in a later packet, not in Closed Autonomy Loop v1.

## How This Advances AI_OS Autonomy

The new loop gives future worker-dispatch integration a structured artifact to
consume. It makes the next action explainable by tying it to evidence, governor
selection, proposed packet shape, gate status, and hard safety boundaries.

The useful progression is:

```text
Evidence -> Governor Decision -> Closed Loop Recommendation -> Future Gated Dispatch
```

## What Remains After v1

Remaining work:

- align downstream consumers on the sandbox latest-report path where needed
- add explicit Start/Pause/Resume/Stop controls
- connect lock and worker-state readiness
- define durable output retention for closed-loop reports
- add approval-aware APPLY dispatch gates
- keep live trading, broker execution, credentials, real webhooks, and schedulers
  blocked until separately approved

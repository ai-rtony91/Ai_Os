## Daily Earned GitHub Work Policy

Daily GitHub work must be legitimate validated repo work. It can come from scoped docs, tests, contracts, evidence adapters, or implementation packets that move AIOS or the forex proof forward.

Green boxes are a byproduct of legitimate validated repo work, not the goal by themselves.

AIOS must never create fake commits, meaningless whitespace commits, generated noise commits, or report churn to manufacture activity.

## Bored Queue Policy

AIOS may use bored queue tasks only when no higher-priority active packet exists and only inside approved safety boundaries.

Good bored queue work includes:

- small docs polish
- test hardening
- repo hygiene docs
- compact dashboard improvements
- forex roadmap acceptance criteria
- paper-only forex evidence expansion
- risk gate docs/tests
- daily contribution loop prep

Bored queue tasks are candidate data only. They do not execute, stage, commit, push, merge, dispatch workers, mutate queues, mutate approvals, or write reports by themselves.

1. Canonical spec.
2. Local data schemas.
3. Deterministic local backtest harness.
4. Risk gate contract.
5. Dashboard/reporting contract.
6. Paper-forward simulator only after separate approval.
7. Broker integration only after separate future protected approval.

The #740 Supertrend proof is paper-only research evidence. It is not a profitability claim and it is not live-ready.

## Protected Live-Readiness Boundary

AIOS must not automatically perform broker integration, OANDA/live exchange integration, live orders, paper orders unless separately approved, credential/secret/env reads or writes, webhooks, scheduler/daemon execution, real-money trading, account mutation, or network market automation.

Live trade readiness requires paper-forward evidence, risk evidence, operator review, protected approval, and future gated implementation work.

## Idle Behavior

When idle, AIOS should first check for active approved packets, selected roadmap candidates, blocked review work, and safe next actions. If none exist, AIOS may surface one bored queue candidate that can produce real repo value inside safety boundaries.

When unsure, AIOS should stop and report the blocker instead of inventing work.

## Never Automatic

AIOS should never automatically:

- trade live or paper orders
- connect to brokers or live exchanges
- load credentials or secrets
- read or write `.env` files
- call webhooks
- start schedulers or daemons
- dispatch workers
- mutate queues or approvals
- write report noise for activity
- stage, commit, push, merge, reset, clean, or delete branches

## Compact Dashboard Relationship

The compact dashboard is the operator stop point. It should show mission, packet, state, progress, next action, bored queue, safety, and detail hint in a short default panel.

Full JSON remains available for inspection, but the default output must stay readable.

## Monthly Wrap-Up Strategy

The month should close with AIOS proving that it can keep moving through safe, validated daily work: completed packet memory prevents repeats, the dashboard stays readable, bored queue work remains legitimate, and forex progress advances only through paper-only and protected stages.

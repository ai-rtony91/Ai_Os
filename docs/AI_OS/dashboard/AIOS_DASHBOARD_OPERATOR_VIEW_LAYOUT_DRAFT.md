# AI_OS Dashboard Operator View Layout Draft

## Purpose

This draft describes what the operator may see first in a future dashboard.

No dashboard output is created in this stage.

## First View

The operator should first see:

- top system status bar
- protected-file status
- validator status
- daily analytics
- alerts / emails
- trading readiness boundary
- next safe action

## Priority Ordering

Priority should favor blocked or unsafe states first, then validator state, protected-file state, daily analytics, Morning Brief, cloud status, and next safe action.

## Alert Severity Hierarchy

Future alert severity may include:

- FAIL or BLOCKED
- WARN or REVIEW REQUIRED
- PASS or READY FOR REVIEW
- INFO

## Future Keyboard Shortcuts And Hotkeys

Future keyboard shortcuts/hotkeys may support panel search, validator refresh, next safe action focus, alert navigation, and layout reset.

Shortcuts are not implemented in this stage.

## Future Panel Pinning And Docking

Future panel pinning/docking may support pinned alerts, pinned protected-file state, docked validator status, docked daily analytics, and expanded Morning Brief.

Panel pinning and docking are not implemented in this stage.

## Future Multi-Monitor Expansion Ideas

Future multi-monitor expansion may support one display for analytics, one display for alerts and readiness, and one display for workflow/Morning Brief review.

Multi-monitor support is conceptual only in this stage.

## Stack Selection Boundary

Future dashboard stack must be selected later based on performance, maintainability, local/offline support, visual quality, security, and operator usability.

Future rendering stack remains TBD.

## Approval Boundary

Dashboard production outputs require separate approval.

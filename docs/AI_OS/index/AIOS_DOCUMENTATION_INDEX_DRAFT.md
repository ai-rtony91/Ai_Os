# AI_OS Documentation Index Draft

## Purpose

This file is a draft index for AI_OS documentation. Its purpose is to help operators, maintainers, and AI coding agents locate current planning, governance, validation, readiness, and roadmap documents without treating this draft as a replacement for the existing authoritative files.

This file is an index draft, not the source of truth yet. Protected root files remain authoritative until explicitly promoted by human approval and documented project governance.

## Authority Boundary

Protected root files remain authoritative for mission, governance, risk, deployment, requirements, source evidence, error logging, hallucination logging, daily reporting, and after-action review. This draft does not override protected root files and does not approve edits to them.

## Documentation Categories

### Dashboard

Dashboard documentation covers future operator cockpit concepts, static preview planning, dashboard readiness, visualization boundaries, and dashboard output restrictions. Dashboard work remains planning-first and must not activate production dashboard writers without approval.

### Telemetry

Telemetry documentation covers future telemetry previews, telemetry readiness, persistence planning, fixture boundaries, and safety rules for non-live telemetry data. Future telemetry outputs remain blocked until explicit approval.

### Metrics

Metrics documentation covers planned health, readiness, validation, and project progress measurements. Metrics must remain non-secret, non-broker, and non-trading-execution data unless separately approved through governance.

### Writers

Writer documentation covers controlled writer concepts, DRY_RUN/APPLY separation, output boundaries, and future promotion planning. No writer is approved for live automation by this index draft.

### Reporting

Reporting documentation covers daily progress outputs, health checkpoint reports, validator summaries, and non-live project status reports. Reports must preserve source evidence and avoid unsupported claims.

### Approval

Approval documentation covers human approval requirements before APPLY stages, protected-file edits, production automation, writer activation, telemetry persistence, dashboard output creation, startup automation, credential access, and any trading-related execution path.

### Router

Router documentation covers future routing concepts for local AI_OS workflows, operator handoff, and task coordination. Routing documentation does not approve broker routing, trade routing, or live execution.

### Morning Brief

Morning Brief documentation covers planned operator summaries and non-live readiness views. Morning Brief output must remain advisory and must not trigger trading, broker actions, or hidden automation.

### Readiness

Readiness documentation covers validation gates, project stage checks, blocked action lists, dry-run discipline, and criteria for future controlled promotion.

### Operations

Operations documentation covers local-first AI_OS operation, checkpoint handling, project folders, operator procedures, and safe execution boundaries.

### Runbooks

Runbooks document repeatable operator procedures. Runbooks must include expected result, stop condition, and approval boundaries for any step that could modify files or system state.

### Audits

Audit documentation covers project reviews against governance, industry-standard engineering practices, safety boundaries, validator coverage, telemetry readiness, dashboard readiness, operator workflow, source control hygiene, production automation boundaries, and trading execution separation.

### Roadmap

Roadmap documentation covers future stage planning and checkpoint criteria. Roadmaps do not approve live automation, trading automation, or APPLY behavior without human approval.

### Index

Index documentation covers maps, ownership references, documentation grouping, and navigation aids. Index files are draft coordination aids until explicitly promoted.

# Phase 14.11 Paper Route Replay Connector

## What Replay Means

Replay means AI_OS takes one saved paper example and walks it through the same local review path again. Nothing is sent outside the computer. The replay is a way to prove that the route can be read, validated, scored, and summarized before any future automation is considered.

## Replay Architecture

Replay path:

1. Load the TradingView-style example alert.
2. Normalize it into the AI_OS paper signal intake shape.
3. Run the AI_OS paper validation check.
4. Build the TradersPost-style blocked handoff payload shape.
5. Calculate deterministic trader execution-quality metrics.
6. Write one scorecard-ready paper replay result JSON.

## Why Replay Matters

Replay gives AI_OS a repeatable proof trail. Instead of guessing whether a signal was handled correctly, the operator can inspect one final replay result that shows the signal, validation status, blocked handoff status, and execution-quality values.

## Why Execution Quality Matters

Execution quality is the paper-only check for expected fill price, actual paper fill price, spread estimate, slippage estimate, and fill latency. These values help show whether a future paper route is worth reviewing. They do not place trades.

## Why Latency Matters

Latency matters because slow signals can become stale. Phase 14.11 records fill latency as a deterministic local value so later stages can compare replay timing and route quality without creating a live order path.

## Why Scorecard-Ready Output Matters

The replay result includes `scorecard_ready` so later paper scorecard work can decide whether the replay result is ready to be included in review. This keeps the scorecard input explicit and easy to audit.

## Paper-Only Scope

The connector runs on local files only. It uses deterministic sample values for fill price, latency, and execution-quality scoring.

Required output safety fields remain fixed:

- `mode`: `paper_only`
- `live_execution_status`: `BLOCKED`
- `broker_status`: `NOT_CONNECTED`
- `traderspost_handoff_status`: `NOT_SENT`

## Replay Route Explanation

The replay result summarizes one mocked BUY route for `EUR_USD` on `M15`. The expected and actual paper fill prices match, the simulated fill latency is zero, and the execution-quality score is generated by the trader execution-quality module.

The output file is:

`apps/trading_lab/mock-data/tv_tp_bridge/paper_route_replay_result.example.json`

## Safety Boundaries

- Local paper replay only.
- No external delivery.
- No credential handling.
- No account connection.
- No trade placement.
- No security, login, SSO, dashboard, or UI changes.

Live execution remains `BLOCKED`. Broker routing remains `NOT_CONNECTED`. TradersPost-style handoff remains `NOT_SENT`.

Phase 14.12 can consume the replay JSON as a scorecard input, compare expected and actual paper fill fields, and add review tags for latency, slippage, validation status, and blocked handoff status.
